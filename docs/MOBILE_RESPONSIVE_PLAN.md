# スマホ・タブレット対応（方針案・承認用）

現在の構成は Bootstrap グリッド＋固定幅グラフ（高さ 320px）で、PC 向けレイアウトです。ここでは **OS ごとに別プログラムを書くのではなく**、ブラウザの共通仕様（CSS メディアクエリ／ビューポート）でまとめて対応し、崩れだけをピンポイントで防ぐやり方を推奨します。

---

## 1. まず決めること（妥協との線引き）

| 項目 | 理想 | 現実的な妥協 |
|------|------|----------------|
| スマホ横持ちの高さ | 全部一度に見える | **高さが極端に低い**。コントロールを折りたたむ／2段構成にして **グラフを優先** |
| ボタンの文言 | 全文常時表示 | 狭い幅では **短いラベル** または **アイコン＋ツールチップ**（タップ長押しは弱いので、字幕は別途検討） |
| iOS と Android で「見え方が 1px も同じ」 | 完全一致 | **ほぼ同じ**にすることは可能。フォント・アンチエイリアス・Chrome/Safari の差で **わずかな差は残る**（許容が現実的） |
| タブレット | PC と同じ余白 | 幅に合わせ **max-width** とカラム数を増やす。縦横で **同じブレークポイント設計** でよい |

**結論:** 完全に「全部常時表示で破綻ゼロ」は難しい。**「操作に必要なものは必ず押せる」「グラフは読める」** を最優先にし、狭いときは **折りたたみ・順序の入れ替え・高さの可変** で妥協するのが妥当です。

---

## 2. 技術方針（おすすめの並び）

### 2.1 ブレークポイント設計（OS 別ではなく「幅・向き・高さ」別）

「iOS 用」「Android 用」の CSS を分けるのではなく、次のような **連続した区切り** が保守しやすいです。

- **narrow phone** … 幅 〜576px 前後（縦持ちスマホ）
- **wide phone / small tablet** … 577px〜768px（横持ちスマホ〜小型タブレット）
- **tablet** … 769px〜（または Bootstrap の `md` / `lg` に寄せる）
- **`orientation: landscape`** … 高さが足りないとき専用の **コンパクトモード**

高さだけ見るクエリ（`max-height`）も併用すると、**横持ち iPhone でグラフが潰れる**問題に効きます。

### 2.2 レイアウトの骨格（HTML は微修正、CSS が主戦場）

- いまの `.row` / `.col` を活かしつつ、狭いときは **`flex-column` に切り替え**・ボタン行を **`w-100`** などでフルワイドに。
- 「ツールバー（マイク・基準・目標・ボタン）」と「グラフ」を **順序固定**でスタック。**横持ちでは「グラフ優先」の並び順** に変える案もあり（スクロール少なく見せる）。
- **`safe-area`**（ノッチ／ホームバー）：`viewport-fit=cover` と `padding-bottom: env(safe-area-inset-bottom)` で **隠れ・誤タップ**を防ぐ。

### 2.3 グラフ（Canvas）

- 幅は **`100%`**（現状どおり）でよい。
- 高さは **固定 320 のままだと横持ちで画面を食う**ため、`min-height`＋**`clamp()` または JS で `resize` 時に高さ算出** のどちらかを採用する案を推奨。
  - **CSS のみ**: `height: min(40vh, 320px)` のように上限・下限を決める。
  - **JS 併用**（現状 `graph.js` で高さ設定あり）: `innerHeight` とヘッダー分を引いた残りを Canvas に渡す。**再現性が高い**。

### 2.4 ボタン・フォーム（iOS / Android での「見え方」）

OS 検出してスタイルを分けるより、**タッチ共通ルール**を先に満たすのが安全です。

- **タップ領域** およそ **44×44px 以上**（Apple HIG）／ **48×48dp 相当**（Material）を目安に。`btn-sm` のままだと小さめなので、狭い画面では **`btn` 相当＋`min-height`**。
- **ネイティブコントロールの差**（`select` の矢印・角丸）は、**Bootstrap の `form-select` のまま許容**するか、`appearance` で多少寄せるかは **優先度低**でよいです（完全同一はコストが高い）。
- iOS で **入力フォーカス時に勝手にズーム**される問題があるため、入力のフォントサイズは **16px 以上**にするのが無難（妥協しないと体感が悪い）。

## 3. OK をもらいたい「採用案」（A と B）

どちらかを選んでください。実装は OK 後に進めます。

### 案 A：CSS 中心（変更が軽い・まず試す）

- `app.css` にメディアクエリを増やす。Bootstrap のユーティリティクラスを `index.html` に少量追加。
- グラフ高さは **`clamp()` ＋ vh** でなんとなく収める。
- **メリット:** 実装が速い。**デメリット:** ごく特殊な機種サイズでは微調整が必要になりうる。

### 案 B：CSS ＋ グラフ高さは JS で計算（おすすめ・破綻しにくい）

- ヘッダー・フォームブロックの高さを確保したうえで、残り領域を Canvas に渡す（`PitchGraph.resizeCanvasToDisplay` を拡張）。
- 向き変更（`orientationchange` / `resize`）でも再計算。
- **メリット:** 横持ちの「グラフだけ見たい」に強い。**デメリット:** `graph.js` / `app.js` に少し手を入れる。

**推奨:** 長期的には **案 B**。短期で様子見なら **案 A → 不満が出た箇所だけ B に寄せる** でもよいです。

## 4. 検証のしかた（実機がなくても最低限）

- Chrome DevTools の **デバイスモード**（幅・ピクセル比・向き）。
- Safari の場合は **実機または Mac + シミュレータ** が確実（一部挙動はエミュレータだけでは足りない）。
- **「フォールバック」**として、極端に小さい高さでは **縦スクロールで全部届く**ことを必須条件にすると安心です。
